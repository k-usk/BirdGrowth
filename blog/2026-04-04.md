# データ永続化の基盤を整備した一日——SwiftData, UserDefaults, そして図鑑JSONへの移行

## 作業内容

本日はアプリの「記憶」を持たせることを主題に、3種類のデータ管理機構を実装した。

### 1. 設計の整理

まず「何を、どう持つか」を整理するところから始めた。要件は以下の4点だった。

- 鳥画像は従来通り `Sprites/` フォルダで管理（変更なし）
- 説明文は鳥のファイル名に紐づいたマスターデータとして管理
- 育てた記録（どの鳥、何日、何歩）の永続化
- 現在育てている鳥の状態（種類、バリエーション、歩数）の永続化

議論を経て、以下の方針で決着した。

| データ | 方法 |
|---|---|
| 鳥画像 | Sprites フォルダ（変更なし） |
| 説明文 | `BirdCatalog.json`（バンドルリソース） |
| 現在の鳥 | UserDefaults + Codable |
| 卒業記録 | SwiftData |

「現在の鳥」に SwiftData を使わなかった理由は単純で、常に1件しか存在しない状態データにコレクション管理のフレームワークは過剰だからだ。

### 2. BirdCatalog.json の作成

`BIRD_ENCYCLOPEDIA.md` に記述されていた120種分のフレーバーテキストを、アプリが機械的に読み取れるJSON形式へ変換した。Pythonスクリプトで抽出し、文中の改行は `\n` として埋め込んでいる。

```json
"1_さすらいのホリドリ": {
  "description": "空を捨て、大地を揺り籠としたさすらいの雛。\n背負いしリュックには、分かち合った数多の幸福が詰まっている。\n彼らが空を見上げるのを止めた理由は、もはや知る由もない。"
}
```

キーはSpritesフォルダのファイル名から `.png` を除いたもの（例: `1_さすらいのホリドリ`）とし、画像・説明文・ViewModel がすべてこのキーで連携する設計にした。

### 3. BirdCatalogLoader の実装

JSONの読み込みは起動時に一度だけ行い、以後はキャッシュを返す構造にした。

```swift
struct BirdCatalogLoader {
    private static var _catalog: [String: CatalogEntry]?

    private static var catalog: [String: CatalogEntry] {
        if let cached = _catalog { return cached }
        guard
            let url = Bundle.main.url(forResource: "BirdCatalog", withExtension: "json"),
            let data = try? Data(contentsOf: url),
            let decoded = try? JSONDecoder().decode([String: CatalogEntry].self, from: data)
        else {
            _catalog = [:]
            return [:]
        }
        _catalog = decoded
        return decoded
    }

    static func description(for spriteKey: String) -> String? {
        catalog[spriteKey]?.description
    }
}
```

シングルトンや `@Observable` を使わず、純粋なstatic関数として実装した。依存が少なく、テストも書きやすい。

### 4. CurrentBird（UserDefaults永続化）

現在育てている鳥の状態を表す `CurrentBird` 構造体を定義した。`Codable` に準拠させることで、UserDefaultsへのエンコード・デコードを簡潔に記述できる。

```swift
struct CurrentBird: Codable {
    var spriteKey: String
    var colorRowIndex: Int
    var startDate: Date
    var steps: Int
}
```

BirdViewModelの `steps` プロパティの `didSet` で `saveCurrentBird()` を呼ぶことで、歩数が増えるたびに自動保存される。

### 5. BirdRecord（SwiftData永続化）

卒業した鳥の記録は `@Model` クラスとして定義した。

```swift
@Model
final class BirdRecord {
    var spriteKey: String
    var colorRowIndex: Int
    var startDate: Date
    var graduationDate: Date
    var totalDays: Int
    var totalSteps: Int
}
```

卒業処理 `graduateBird(context:)` は `ModelContext` をView側から受け取る形にした。ViewModelがSwiftDataに直接依存しすぎないよう、Contextは外から注入する設計にしている。

### 6. BirdViewModel の刷新

最も変更量が多かった箇所だ。従来は起動のたびにランダムで鳥を選択していたが、今回から `setupSprites()` でUserDefaultsの状態を優先復元するよう変更した。

```swift
func setupSprites() {
    let urls = Bundle.main.urls(forResourcesWithExtension: "png", subdirectory: "Sprites") ?? []
    availableSprites = urls

    if let data = UserDefaults.standard.data(forKey: .currentBirdKey),
       let saved = try? JSONDecoder().decode(CurrentBird.self, from: data) {
        currentBird = saved
        steps = saved.steps
    } else {
        Task { await startNewBird() }
    }
    // ...
}
```

### 7. 図鑑機能の実装（EncyclopediaView）

SwiftData に蓄積された `BirdRecord` を一覧表示する図鑑機能を実装した。`LazyVGrid` を採用し、3カラムのグリッド形式で成鳥のイラストを並べている。

タップ時には詳細画面（`BirdDetailSheet`）が開き、育てた日付や総歩数などの統計を確認できる設計にした。

### 8. UI/UX のブラッシュアップ

「鳥をより愛おしく感じさせる」という指針に基づき、細かな表示制御を徹底した。

- **成長段階の振り返り**: 詳細画面に `Picker` を追加し、たまご・ひな・成鳥の姿を自由に切り替えられるようにした。
- **レイアウトの固定**: 1万歩達成時に出現する導線によってイラストが上下にズレないよう、固定の余白をあらかじめ確保する構造に変更。
- **即時切り替え**: 成長時やバリエーション変更時のスプライトの「スライド演出」を廃止。`.animation(nil)` を適用し、パッと切り替わる潔い操作感を実現した。
- **テキスト整形**: 伝承（Lore）の視認性を高めるため、20文字ごとの強制改行ロジックを実装し、左寄せ配置に変更した。

## 苦労した点

### JSON変換スクリプトの出力が空になる問題

最初の変換スクリプトでは、正規表現が `>` で始まる引用ブロックをうまく拾えず、全120エントリの `description` が空文字になった。原因は、Markdownの `>` 記法がスクリプトの正規表現パターンと噛み合っていなかったことだ。

解決策として、`### No.` で文字列を分割してセクション単位で処理する方式に変更し、各行の先頭文字を確認することで確実に引用行だけを抽出するようにした。

### 図鑑一覧でのタップ判定バグ（ヒットテストの重複）

`scaleEffect` と `offset` を多用してスプライトを切り出している `BirdGridItem` において、拡大された画像の「不可視領域」が隣のセルのタップを横取りするバグが発生した。

SwiftUI の `.clipShape` は視覚的には切り取るが、タップ判定領域までは制限しない。解決策として、画像自体に `.allowsHitTesting(false)` を付与し、セルの親コンテナに `.contentShape(Rectangle())` を設定することで、タップ判定を表示範囲（80x80）に厳密に閉じ込めた。

### 透明なデバッグメニューの視認性

デバッグメニューを `.presentationBackground(Color.white.opacity(0.2))` で透過させたが、今度はボタンが背景に溶け込んで見えなくなる問題が起きた。不透明度を `0.7` まで引き上げ、デザイン性と実用性のバランスを調整した。
