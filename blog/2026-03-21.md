# スマホからのAI自動開発：GitHub Actions×GeminiによるPR作成フロー構築とAPI制限の壁

本日は、SwiftUIベースのiOSアプリ「BirdGrowth」におけるUIコンポーネントの調整と、Issue駆動によるGemini APIの自動コード生成・PR（Pull Request）作成パイプラインの構築を実施した。特に後者のCI環境構築において、APIのレートリミットや権限付与、セキュリティなど複数の課題に直面し、これらを解決した。

## 本日の作業内容

1. **SwiftUIコンポーネントの非活性・初期段階UIの調整**
2. **AI開発アシスタント用ルールファイル（.agent/rules.md等）の統合**
3. **GitHub Actionsを用いたIssueドリブンな自動実装・PR生成インフラの構築**

## 1. SwiftUIコンポーネント（ContentView）のUI微調整
卵アイコン（`GrowthStage.egg`）の視認性およびデザイン性を高めるため、色彩調整および `.shadow` 修飾子を利用して浮遊感を付与した。

```swift
// ContentView.swift (抜粋)
.shadow(
    color: stage == .egg
        ? Color.black.opacity(0.25)
        : Color.clear,
    radius: 6,
    x: 0,
    y: 18
)
```
**解説:**
単純に `Color.black` を使用するのではなく `.opacity(0.25)` で不透明度を抑え、影の重心をY軸方向にずらすことで（`y: 18`）、要素がUIレイヤーから物理的に浮き上がっているような表現を加えた。要件に応じた細かい修飾子の適正値を見つけるプロセスは、SwiftUIにおけるUI実装の基本となる。

## 2. 自動実装パイプラインの構築と直面した課題

本日の最大の難所は、**「スマートフォンからGitHub上にIssueを立てた際、AIが自動でコードを書き換え、該当のソースファイルのみを含む検証可能なプルリクエストを生成する」**という一連のフローの開拓であった。

初期アプローチとして、Googleが提供する `gemini-cli` をGitHub Actions内で呼び出す自動化を検証したが、以下の技術的な壁に直面した。

### 課題A: Gemini APIのFreeプランにおけるRate Limit超過
`gemini-cli` のような自律型エージェントツールは、リポジトリ内のディレクトリ構造やコンテキストを把握するために、バックグラウンドで並列に多数のリクエストを送信するアーキテクチャとなっている。そのため、Freeプランにおける「1分間あたり15〜20リクエスト」という制限（RPM）を即座に超過し、`Quota exceeded` エラーで処理が停止する事態が発生した。

### 課題B: 実行環境特有の権限エラー（リスト取得へのブロック）
CI環境のサンドボックス制限により、エージェント内部でコールされる `list_dir` コマンドがブロックされた。これにより対象のコードファイルが特定・編集されず、「Actionの実行自体はSuccess（成功）だが、ファイルの変更差分が0件であるため、空のPRすら生成されない」という現象を引き起こした。

### 課題C: サードパーティによる不正実行（悪用）リスク
公開リポジトリ（Public）においてIssueのコメントをトリガーとする場合、悪意のある第三者が無限にスクリプトを呼び出し、自身のAPIクレジット枠やActionsの実行枠を枯渇させることが理論上可能であった。

## 3. 解決策：カスタムスクリプトの直接実行への移行

これら全ての課題をクリアするため、高機能だが処理がブラックボックス化された `gemini-cli` の利用を見送り、Actions内で独自に `.js` スクリプトを生成・実行する設計に切り替えた。

以下は、本日最終的に構築した `gemini-solve.yml` の要所抜粋である。

```yaml
jobs:
  gemini_solve:
    # 悪用防止対策：リポジトリオーナーの特定コマンド発下時のみ動作させる
    if: >
      startsWith(github.event.comment.body, '/gemini') && 
      github.event.comment.user.login == github.repository_owner
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
    steps:
      # ... (checkoutやNodeのセットアップ処理を省略) ...
      
      - name: Run Custom Gemini Script
        run: |
          cat << 'EOF' > run_gemini.js
          const fs = require('fs');
          const { GoogleGenerativeAI } = require('@google/generative-ai');
          
          // APIに投げるファイルスコープを絞り、1回の通信で完結させる
          const filePath = 'BirdGrowth/ContentView.swift';
          const fileContent = fs.readFileSync(filePath, 'utf8');
          
          // (略) genAIを通して gemini-2.5-flash で1回だけ推論を実行
          
          // Markdownの不要なコードブロックを正規表現で切り落として上書き保存
          text = text.replace(/```swift\n?/i, '').replace(/```\n?/g, '').trim();
          fs.writeFileSync(filePath, text);
          EOF
          node run_gemini.js

          # 【重要】PRに余計なライブラリが混入するのを防ぐ削除処理
          rm -rf node_modules package.json package-lock.json run_gemini.js
```

### 解決のポイント解説

1. **API制限の確実な回避**
   必要なファイル（ここでは `ContentView.swift`）を明示的に指定し、プロンプトに文字列として直接埋め込む手法を取った。これによりGeminiモデルとの通信は「1回のみ」で確定し、Freeプランの厳しいRate Limit環境下でも安定稼働を実現した。
2. **実行権限の制御とセキュアな設計**
   `github.repository_owner` と一致する書き込みのみを許可し、さらに呼び出し文字列を `@gemini-cli` から `/gemini` に変更した。これにより、実在する他ユーザーへの無用なメンション通知を防ぐと共に、管理者以外のキックをシャットアウトした。
3. **不要ファイルの混入防止による綺麗なPRの生成**
   スクリプト内で使用するために一時ダウンロードした `node_modules` や、スクリプト実行ファイル本体の破棄（`rm -rf`）を、PR生成アクションに移行する直前に行っている。自動化スクリプト自体の残骸を取り除くことで、本来のアプリのコードのみが差分に現れる仕様とした。
4. **GitHub設定における権限の開放**
   Actionsのymlに `permissions` を記載するだけでは完結せず、リポジトリ設定（Settings > Actions > General > Workflow permissions）にて「`Allow GitHub Actions to create and approve pull requests`」を有効化しないとプルリクエスト作成段階でエラーとなる点にも留意が必要である。

総括として、AIを用いた作業自動化においては、便利で高機能な外部エージェントツールの挙動を完全に制御しきれない場面で多くの罠が存在する。要求水準を満たす範囲で自前のシンプルな単一スクリプトに切り替える実装は、デバッグの容易さと保守性、そしてコストの観点で非常に有効な手段であった。
