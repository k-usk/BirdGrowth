# Swift Testing導入とGitHub CI構築：スキーム保存の落とし穴

本日は「BirdGrowth」のコード品質向上を目指し、Viewのリファクタリング、自動テストの導入、およびGitHub ActionsによるCI環境の構築を行った。特に、スキームファイル（`.xcscheme`）の挙動に関する技術的課題とその解決策について記録する。

## 作業内容
1. **Viewのリファクタリング**: `ContentView` から背景描画 (`BackgroundView`) とデバッグ操作 (`DebugControlsView`) をコンポーネントとして分離。
2. **自動テストの導入**: `Swift Testing` によるロジック検証と `ViewInspector` によるUI構造検証を実装。
3. **CI環境の構築**: GitHub Actions (`ci.yml`) を用い、プルリクエスト時に自動で `SwiftLint` と全テストを実行する仕組みを構築。

## 実装したテストケース
`BirdViewModel` の成長ロジックを中心に、以下のケースを網羅した。

- **成長境界値テスト**: 4,999歩（卵）、5,000歩（ひな）、9,999歩（ひな）、10,000歩（成鳥）の各境界値でステージが正確に遷移するか。
- **メッセージ抽選**: 歩数セグメント（1,000歩単位）の更新に合わせ、適切なメッセージプールからランダムに選択されているか。
- **データ解析**: 画像ファイル名（例: `blue_parrot#1.png`）から、表示用の正規化された名前（blue parrot）が正しく抽出されるか。
- **UI表示**: `ViewInspector` を用い、現在の歩数やステータスメッセージが正しく `Text` として配置されているか。

## 苦労した点：*.xcscheme の不在とCIエラー
GitHub Actionsでのテスト実行 (`xcodebuild test`) において、「スキームが見つからない」「テスト用デバイスが見つからない」というエラーに直面した。

### *.xcscheme ファイルの意味
`*.xcscheme` （スキームファイル）は、Xcodeに対して「どのターゲットを、どういう設定（Debug/Release等）でビルド・テスト・実行するか」を記述した指示書である。通常、これは個人の設定（`xcuserdata`）としてGit管理外に置かれるが、CIで利用するためには「Shared」設定を有効にして物理ファイルを `xcshareddata` フォルダに出力し、Gitにコミットする必要がある。

### 発生した問題と解決策
Xcode上で「Shared」にチェックが入っているにもかかわらず、物理ファイルがディスク上に生成されない現象が発生した。これはXcodeの自動生成（Autocreate）機能に依存している場合に起こる。スキームの「Edit」画面を一度開き、何も変更せずに閉じる操作を強制することで物理ファイルを生成させ、さらに不足していた `TestAction` の定義をXMLへ手動追記することで、CI環境での正常動作を実現した。

## コードの要所解説
### Swift Testing による境界値検証
```swift
@Test("歩数に応じた成長ステージの遷移確認")
func testStageTransitions() {
    let viewModel = BirdViewModel()
    viewModel.steps = 4999
    #expect(viewModel.stage == .egg)

    viewModel.steps = 5000
    #expect(viewModel.stage == .chick)
}
```
`Swift Testing` の `#expect` マクロを使用。従来の `XCTest` よりも宣言的で読みやすく、エラー時の情報量も豊富である。

### GitHub Actions における環境固定
```yaml
runs-on: macos-15
steps:
  - name: Select Xcode 16.2
    run: sudo xcode-select -s /Applications/Xcode_16.2.app/Contents/Developer
```
`macos-latest` のデフォルト環境に依存せず、`macos-15` と `Xcode 16.2` を明示的に指定。iOS 18.2 ターゲットおよび `Swift Testing` の動作要件を確実に満たす構成とした。

## アーキテクチャの妥当性
MVVMパターンに基づき、Viewから計算ロジックを完全に分離したことで、今回のような大規模なテスト導入をスムーズに行うことができた。また、GitHub上でのPRベースの開発フローにCIを統合したことで、今後は「テストが通らなければマージできない」という強力な品質ガードレールが機能する。
