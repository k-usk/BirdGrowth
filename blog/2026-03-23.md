# Xcode「幽霊フォルダ」との決別：Assets方式の挫折を経てFolder Reference完全自動化へ至った死闘の記録

昨日（2026-03-22）の作業終了時点、我々は絶望の中にいた。`Bundle.main.urls` で画像を取得しようとしても結果は常に「0枚」。シミュレータ上では鳥は一羽も見当たらず、原因不明のまま「WIP（作業中）」として筆を置くしかなかった。本日は、その絶望をどう突破し、最終的に「画像をフォルダに置くだけ」という全自動化の理想郷へ辿り着いたか、その全プロセスを記録する。

## 第1戦：`.app` バンドルの解剖と「幽霊フォルダ」の正体

まず着手したのは、昨日失敗した原因の特定だ。Xcodeのプロジェクトナビゲータ上では確かに `Sprites/` フォルダに画像が存在している。しかし、実行時に読み込めない。ならば「そもそもビルド後のアプリ内に画像が入っていないのではないか？」という仮説を立て、ターミナルでビルド生成物（`.app`）を直接解剖した。

```bash
ls -la /Path/To/BirdGrowth.app/
```

驚愕の事実が判明した。バンドル内には実行ファイルと `Info.plist` しかなく、`Sprites` フォルダの影も形もなかったのだ。
原因は、Xcode 16の新機能 **`PBXFileSystemSynchronizedRootGroup`**（外部フォルダとの自動同期）にあった。この機能はSwiftファイルなどは自動で認識するが、PNGなどのフォルダ構造を維持したリミテッドなリソースコピーには、標準の設定だけでは対応しきれない「見えない壁」が存在していた。

## 第2戦：`Assets.xcassets` への回帰と、見えてきた「運用の罠」

「フォルダがコピーされないなら、Xcodeが最も得意とする `Assets.xcassets`（アセットカタログ）に乗せるしかない」。そう判断した私は、一旦「フォルダスキャン」を諦め、すべてのPNGを `.imageset` 形式に変換してアセットカタログへ統合する方針に切り替えた。

スクリプトを組み、数百枚の画像セットを自動生成する準備を整え、`ContentView.swift` を `Image("Sprites/画像名")` で読み込む方式へ全面刷新。この時点では「これで解決だ！」と胸を張っていた。

しかし、実装を進めるうちに致命的な欠点に気づく。
- 新しい画像を1枚追加するたびに、変換スクリプトを走らせる必要がある。
- さらに、コード内の画像名リスト（Array）を毎回手動で更新しなければならない。

これでは「画像を放り込むだけでいい」という当初の理想とは程遠い。解決したかに見えて、実は「運用の地獄」への入り口に立っていたのだ。

## 第3戦：逆転の「Folder Reference」。Qiitaに学んだ10年来の正攻法

この閉塞感を打ち破ったのは、ペアプログラミング相手（ユーザー）からの「この方法はどうだろうか？」という一つの提案だった。

- **参考リンク**: [Xcodeでフォルダをそのままの構成でBundleに含める方法 (Qiita)](https://qiita.com/tunaingot/items/76169242c404b03a8818)

そこに記されていたのは、Xcode 16の同期グループのような新機能ではなく、古くからある **「Folder Reference（フォルダ参照／青いフォルダ）」** という仕組みだった。

### 覚醒した「完全自動スキャン」
これまでの「仮想的なグループ」としてのフォルダ管理を捨て、Xcodeにフォルダを「実体としての参照（青いフォルダ）」として登録し直した。その瞬間、すべてが噛み合った。

1. **Bundleに実体が出現**: ビルド後の `.app` 内に、ついに物理的な `Sprites/` フォルダがコピーされるようになった。
2. **コードの全自動化**: `Bundle.main.urls` によるフォルダ内全スキャンが、ついに火を吹いた。

```swift
private func setupSprites() {
    // フォルダの中身を全部よこせ！
    let urls = Bundle.main.urls(forResourcesWithExtension: "png", subdirectory: "Sprites") ?? []
    availableSprites = urls
    currentSpriteURL = urls.randomElement()
}
```

この実装により、コードを1行も変えることなく、Finderで画像をフォルダに入れるだけで、次の起動時には新キャラがガチャに出現する。これこそが追い求めていた「メンテナンスフリー」の完成形だ。

## 最終調整：ユーザー体験の磨き込み

仕組みが完成した後は、UIの完成度を高める作業に集中した。

- **名前の整形**: ファイル名が `オキナインコ_ブルー#2.png` なら、`_` をスペースに変え、`#` 以降の連番をカットして「オキナインコ ブルー」と美しく表示。
- **成長に応じた開示**: 名前は最初から出さず、成鳥になった瞬間にふわっと出現させる演出。
- **レイアウトの固定**: 名前が現れた瞬間にインコの画像がガタッと下に動かないよう、透明なプレースホルダーで高さを事前に確保する。

## 総括
